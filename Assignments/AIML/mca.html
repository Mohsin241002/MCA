DNS Security Vulnerabilities

Why DNS is Vulnerable

Designed in 1983: When internet was small and trusted
  • No built-in security
  • Plain text communication
  • No authentication

Common DNS Attacks

üåä 1. DNS Flood Attack (DoS)

  • Method: Overwhelm DNS server with massive queries
  • Result: Websites become unreachable
  • Example: 2016 Dyn attack took down Twitter, Netflix

‚ò†Ô∏è 2. DNS Cache Poisoning

  • Method: Inject fake DNS responses into cache
  • Process: Attacker races to respond before real server
  • Result: Users redirected to malicious sites

üïµÔ∏è 3. Man-in-the-Middle via DNS

  • Method: Control DNS server (WiFi hotspots)
  • Result: All traffic routed through attacker

Step 2: Driver Matching Algorithm

The Matching Challenge

Problem: At 6 PM in Bangalore, 10,000 people want rides and 8,000 drivers are available. How do you match them optimally?

Factors Considered (15+ parameters)

üöó Driver Factors:

  • Distance from pickup (primary)
  • Driver rating (4.2, 4.8, etc.)
  • Vehicle type match
  • Driver's acceptance rate
  • Time since last trip

üéØ Trip Factors:

  • User's rating
  • Trip distance
  • Traffic conditions
  • Time of day
  • Historical data

The Algorithm Process

  1. Geofencing: Find drivers within 2km radius
  2. Filtering: Remove busy/offline drivers
  3. Scoring: Calculate match score for each driver
  4. Ranking: Sort by best matches
  5. Sequential Offering: Offer to top 3 drivers

Step 3: Dynamic Pricing & ETA Calculation

Surge Pricing Algorithm

Real-time Demand √∑ Available Supply = Surge Multiplier

üìä How It Works:

  • High Demand + Low Supply = 2x-5x pricing
  • Examples: Airport during flight delays, stadium after match
  • Updates every 2-3 minutes based on live data
  • Heat Maps: City divided into hexagonal zones

ETA Calculation

üßÆ Complex Formula:

ETA = Base Travel Time + Traffic Factor + Driver Behavior + Historical Data

  • Base Time: Google Maps API distance/time
  • Traffic Factor: Real-time traffic multiplier (1.2x, 1.5x)
  • Driver Behavior: This driver typically takes 10% longer
  • Historical Data: This route usually has delays at 6 PM
Accuracy: Uber's ETA is correct within 1-2 minutes 85% of the time

IRCTC System Architecture

Multi-Layer Defense System

üö™ Layer 1: Load Balancers

  • Purpose: Distribute traffic across multiple servers
  • Capacity: Handle 120,000 requests per second
  • Smart Routing: Send users to least busy servers

üè† Layer 2: Web Servers

  • Quantity: 100+ servers running simultaneously
  • Auto-scaling: More servers added during peak times
  • Geographic Distribution: Servers in Mumbai, Delhi, Bangalore

üóÑÔ∏è Layer 3: Database Cluster

  • Master-Slave Setup: Multiple database copies
  • Real-time Sync: All databases stay updated
  • ACID Compliance: Ensures data consistency
Result: System can theoretically handle 10 million users, but deliberately limits access to ensure stability

Queue Management & Fair Booking

The Virtual Queue System

Problem: If 2 million people hit the server at 10:00:00 AM, the system will crash

Solution: Controlled Access

üé´ 1. Token-Based System

  • Users get unique tokens when they login
  • Tokens have priority scores
  • Only limited tokens allowed to book at once

‚è∞ 2. Time-based Releases

  • 10:00:00: First 50,000 users allowed
  • 10:00:30: Next 50,000 users allowed
  • 10:01:00: Next batch, and so on...

üéØ 3. Priority Algorithm

  • Login Time: Earlier login = higher priority
  • User History: Regular users get slight preference
  • Payment History: Users with successful payments prioritized
  • Geographic Load: Balance across regions

Real-time Inventory Management

The Seat Booking Challenge

Scenario: Train has 72 seats. 5000 people are trying to book simultaneously. How do you ensure no double booking?

üîí 1. Pessimistic Locking

  • When someone starts booking, seats get "locked"
  • Other users see "seats not available"
  • Lock released after 10 minutes if payment not completed

‚ö° 2. Atomic Transactions

All-or-Nothing Principle:

  • Either the entire booking succeeds OR it completely fails
  • No partial bookings (half-booked tickets)
  • Database rollback if any step fails

üéØ 3. FIFO with Priority

  • Base Rule: First-come, first-served
  • Priority Exceptions: Senior citizens, disabled passengers
  • Quota System: Different quotas (General, Ladies, Senior)

Anti-Fraud & Security Measures

The Fraud Challenge

Problem: Automated bots, ticket touts, and bulk booking attempts trying to game the system

ü§ñ 1. Bot Detection

  • CAPTCHA: Visual puzzles that humans can solve
  • Behavioral Analysis: Mouse movements, typing patterns
  • Request Patterns: Too many requests = suspicious
  • Browser Fingerprinting: Unique device identification

üé´ 2. Booking Limits

  • Per User: Maximum 6 passengers per booking
  • Per Day: Limited bookings per user per day
  • Per IP Address: Multiple accounts from same IP flagged

üí≥ 3. Payment Verification

  • OTP Verification: SMS/Email confirmation required
  • Card Validation: Real-time fraud detection
  • Payment Time Limit: 10 minutes to complete payment
Result: 90%+ reduction in fraudulent bookings while maintaining user experience

Design Trade-offs

üéØ CAP Theorem

You can only guarantee 2 out of 3:

  • Consistency: All nodes see same data simultaneously
  • Availability: System remains operational
  • Partition Tolerance: Works despite network failures

System Choices:

  • üîó DNS: A + P (Sacrifices consistency)
  • üöÜ IRCTC: C + A (Sacrifices partition tolerance)
